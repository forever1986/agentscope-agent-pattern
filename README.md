# 第二部分 Agent设计模式

这一部分主要是根据《agentic-design-patterns》精讲经典架构模式，掌握高阶 AI 应用设计思路。设计模式包括：

- **提示词链**：也称为管道模式（Pipeline pattern），该模式摒弃了让 Agent 在单一整体化步骤中解决复杂问题的做法，采用分而治之策略：将原始复杂问题分解为更小、更易管理的子问题序列，每个子问题通过专门设计的提示词单独处理，输出作为输入传递给链中后续提示词。
- **并行**：Agent任务涉及多个可同时执行而非顺序执行的子任务，识别工作流中不依赖其他部分输出的环节并并行执行。
- **路由**：将条件逻辑引入Agent系统的工作框架，使其能够从固定的执行路径转变为这样一种模型：智能体系统动态评估特定标准，从一组可能的后续操作中进行选择。这使得系统行为更加灵活且具有上下文感知能力。
- **规划**：Agent自主规划实现目标的路径：先理解初始状态（如预算、参与人数、期望日期）和目标状态（成功预订活动），再设计连接两者的最优行动序列。计划并非预先设定，而是根据请求动态生成。
- **反思**：Agent评估其自身工作、输出或内部状态，并利用该评估来提升性能或优化响应。
- **推理**：Agent能够将复杂问题分解为更小的子问题、考虑中间推理步骤，并得出更加可靠和准确的结论。
- **人机协同**：从根本上围绕Agent与人类的协同理念展开。它并非将 AI 视为人类工作者的替代品，而是将其定位为增强和提升人类能力的工具。
- **异常处理和恢复**：专注于开发异常耐用且有弹性的Agent，使其能够保持不间断的功能和操作完整性，即使面临各种困难和异常情况。
- **Guardrails安全模式**：确保Agent安全、符合道德规范并按预期运行的关键机制，特别是在Agent自主并集成到关键系统中的情况下。
- **Handoffs（交接）**：一种动态委托机制，允许一个 AI Agent 在完成其工作或遇到能力边界时，将控制权、对话上下文和任务状态显式转移给另一个更适合处理当前任务的Agent。
- **Subagents（子智能体）**：Orchestrator（主编排Agent）委托工作的专职Agent。编排Agent不亲自执行这些工作，而是通过 Task 工具传入子智能体类型和任务描述来调用。
- **Supervisor（监督者）**：一个中心监督者（主智能体）通过工具调用专职智能体进行协调。监督者接收用户请求，决定调用哪些专家（如日历、邮件），并将结果综合成一条回复。专职智能体从用户视角看是无状态的；监督者维护对话，将一次性任务委托给专家。
- **多智能体辩论**：是微软提出的一种正式AI框架，其中多个不同的Agent协作和争论以解决问题，超越了单个Agent的”思维链”。
